Section 1 — <title>

The <title> element sets the text shown in the browser tab, bookmarks, and search engine results. It is the most impactful single element for SEO.

Browser tab preview

✎ My Page Title ×
Another Tab
https://example.com/

60 characters remaining

<head>
  <title>My Page Title</title>
</head>

Keep titles under 60 characters — Google truncates longer ones in search results. A good pattern: Page Name – Site Name.

Title in search results

https://hes-so.ch › web-dev
HTML & CSS Course – HES-SO Valais
Interactive demo pages for the Web Development course. Each page covers a core HTML/CSS topic with live examples.
<title>
  HTML &amp; CSS Course &ndash; HES-SO Valais
</title>

The title is the main clickable link in a Google result. It should be descriptive, unique per page, and include relevant keywords naturally.

Dynamic titles in SPAs: In React or other single-page applications, the <title> is not automatically updated on navigation. Use document.title = 'New Title' in JavaScript, or a library like react-helmet, to update it on every route change — required for accessibility (screen reader users rely on the title to know what page loaded).

Section 2 — <meta> Tags

Meta elements provide metadata about the document: character encoding, viewport settings, SEO descriptions, and social sharing cards.

Essential meta tags

charset="utf-8"Supports all characters (emoji, accented letters…)
viewportControls layout on mobile devices
descriptionSnippet shown in search results
authorName of the document author
robotsControls search engine indexing
<meta charset="utf-8">
<meta name="viewport"
  content="width=device-width,
           initial-scale=1">
<meta name="description"
  content="Learn HTML and CSS
           interactively.">
<meta name="author"
  content="Jane Smith">
<meta name="robots"
  content="index, follow">

charset has no name attribute; it uses its own charset attribute. It must appear in the first 1024 bytes of the document.

Open Graph (OG) tags — social preview builder

HES-SO VALAIS
HTML & CSS Course
Interactive demo pages for the Web Development course.
<meta property="og:type"
      content="website">
<meta property="og:title"
      content="HTML &amp; CSS Course">
<meta property="og:description"
      content="Interactive demo pages...">
<meta property="og:site_name"
      content="HES-SO Valais">
<meta property="og:image"
      content="https://...">

Open Graph tags control the preview card shown when a URL is shared on Slack, LinkedIn, Twitter/X, or iMessage. The og:image should be 1200×630 px.

Meta tags are not magic: They are hints, not commands. Search engines may override your description with content they consider more relevant. The keywords meta tag is completely ignored by Google (abused for spam in the early 2000s). Focus on real content quality over meta tag keyword stuffing.

Section 3 — <link>

The <link> element defines relationships between the current document and external resources. It is most commonly used for stylesheets and favicons.

Stylesheet

The most common use: linking an external CSS file.

rel="stylesheet"Declares the relationship type
hrefPath to the CSS file
<link rel="stylesheet"
      href="css/style.css">

Place stylesheets in <head> to avoid a Flash of Unstyled Content (FOUC). The browser blocks rendering until linked CSS is downloaded.

Favicon

The small icon shown in the browser tab and bookmarks.

.icoLegacy format, wide support
.svgScalable, best for HiDPI screens
.pngFallback for specific sizes
<!-- SVG favicon (modern) -->
<link rel="icon"
      href="favicon.svg"
      type="image/svg+xml">

<!-- ICO fallback (legacy) -->
<link rel="icon"
      href="favicon.ico"
      sizes="any">

<!-- Apple home screen icon -->
<link rel="apple-touch-icon"
      href="apple-touch-icon.png">

Provide both SVG (modern) and ICO (legacy) favicons. SVG supports dark mode via prefers-color-scheme media queries inside the SVG.

preload & canonical

rel="preload"Start fetching a resource early, before it is discovered in CSS/JS
rel="canonical"Tell search engines which URL is the "official" version to avoid duplicate content penalties
rel="alternate"Alternate versions: RSS feed, different language, print stylesheet
<!-- Preload critical font -->
<link rel="preload"
      href="font.woff2"
      as="font"
      type="font/woff2"
      crossorigin>

<!-- Canonical URL -->
<link rel="canonical"
      href="https://example.com/page">

<!-- RSS Feed -->
<link rel="alternate"
      type="application/rss+xml"
      href="/feed.xml">

preload with as="font" requires crossorigin even for same-origin fonts, due to CORS requirements for font resources.

Render-blocking: Every <link rel="stylesheet"> blocks rendering until the CSS is downloaded and parsed. This is intentional — the browser needs styles before painting. Use preload for critical resources like hero fonts or images to reduce perceived load time.

Section 4 — <style>

The <style> element embeds CSS directly inside the HTML document, without an external file. It is placed in <head>.

Inline styles vs external stylesheet

<style><link>
LocationSame fileSeparate file
HTTP requestNone+1 request
CacheableNoYes
Reusable across pagesNoYes
Best forCritical CSSGeneral styles
 
<head>
  <!-- Critical CSS inline -->
  <style>
    body { margin: 0; }
    .hero { min-height: 100vh; }
  </style>

  <!-- Rest of CSS deferred -->
  <link rel="stylesheet"
        href="css/main.css">
</head>

Inline <style> is ideal for critical CSS: the minimal styles needed to render the above-the-fold content without any render-blocking requests.

Scoped usage

Common legitimate uses for <style>:

  • Critical CSS inlining for performance
  • Email HTML (no external CSS allowed)
  • Single-file HTML demos and prototypes
  • CMS pages where external files are unavailable
<!-- Email-compatible inline CSS -->
<style>
  .btn {
    background: #4c7daf;
    color: #fff;
    padding: 12px 24px;
    border-radius: 4px;
    text-decoration: none;
    display: inline-block;
  }
</style>

Email clients strip external stylesheets — inline <style> blocks (and even style attributes) are the only reliable way to style HTML emails.

Do not use <style> as a default: For multi-page websites, always prefer external stylesheets. They are cached by the browser after the first visit, meaning subsequent pages load with zero CSS download cost. Inline styles in <style> are re-downloaded on every page load.

Section 5 — <script> in <head>

Scripts placed in <head> must use defer or async to avoid blocking HTML parsing. This section previews the concept — all loading strategies are covered in depth on the Scripts page.

Without defer — blocking

HTML parse
Script 
<head>
  <!-- BLOCKING: HTML parse stops
       until script is fetched
       and executed -->
  <script src="app.js"></script>
</head>

The red zone is the blocked period. Users see a white screen while the script downloads. Avoid this pattern.

With defer — non-blocking

HTML parse
Script
Downloading Waiting for HTML Executing 
<head>
  <!-- NON-BLOCKING: downloads
       in parallel, executes only
       after HTML is parsed -->
  <script src="app.js"
          defer></script>
</head>

With defer, HTML parsing is never interrupted. The script runs in document order, after the DOM is ready. This is the recommended approach for <head> scripts.

Rule of thumb: Always place <script> tags with defer in <head>. This starts downloading the script immediately (while HTML is parsing) but runs it safely after the DOM is built. See the full comparison of defer, async, type="module", and end-of-body placement on the Scripts page.

Section 6 — <base>

The <base> element sets the default URL and target for all relative links and resources in the document. There can only be one per page.

Setting a base URL

<base href="https://cdn.example.com/">
img/logo.png → https://cdn.example.com/img/logo.png
css/style.css → https://cdn.example.com/css/style.css
page.html → https://cdn.example.com/page.html 
<head>
  <base href="https://cdn.example.com/"
        target="_blank">
</head>

<!-- These links now resolve
     relative to the base URL -->
<img src="img/logo.png">
<link rel="stylesheet"
      href="css/style.css">
<a href="docs/guide.html">Guide</a>

The optional target attribute sets the default target for all links on the page — e.g., target="_blank" opens all links in a new tab.

Caveats

Anchor links break<a href="#section"> becomes absolute URL + #section, not a page-local anchor
Only one allowedIf two <base> elements exist, only the first is used
Affects all resourcesImages, links, scripts, and stylesheets are all affected
<!-- With base set, this breaks! -->
<a href="#section">Jump</a>
<!-- Resolves to: https://cdn.example.com/#section -->

<!-- Fix: use full relative path -->
<a href="current-page.html#section">
  Jump
</a>

Anchor links (#id) behave unexpectedly with a <base> element. This is a common trap when migrating to a CDN.

When to use <base>: Rarely. The most legitimate use case is serving static HTML files from a CDN root while keeping relative asset paths in the HTML. For most projects, it is simpler to use proper relative or absolute paths in each element individually.
